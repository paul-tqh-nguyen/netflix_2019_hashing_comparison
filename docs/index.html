Hashing Is Not A Silver Bullet

A case study in how satisfaction with O(1) performance can be insufficient.

Introduction

Very frequently, computer scientists gloss over implementation details that can have huge performance implications that can make or break a product or research idea. For example, a naive computer scientist might say that the choice between merge sort and quicksort is insignificant give their equivalent computational complexity. This is demonstrably false due to the cost of memory allocation in merge sort.

This same sort of mistake happens frequently when people use hash tables. Newer computer scientists are frequently satisfied with the constant time lookup cost of hash tables and don’t go into optimizing beyond that. However, hash table lookups have a non-negligible cost associated with them due to spatial locality. Looking up several items in a hash table frequently requires hopping around several non-adjacent places in memory. Though the lookups are O(1), the constant time impact on performance due to a lack of spatial locality is non-negligible, especially for large inputs.

This article aims to show empirically how hash table lookups are not sufficiently performant by comparing the performance of Dijkstra’s algorithm on graphs represented using hash tables and on graphs using adjacency matrices, which are stored in contiguous blocks of memory.
